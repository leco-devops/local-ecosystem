7 · AI-assisted onboarding

Cloud VM (ai-cloud profile): External LLM APIs are the recommended path. Set OpenAI, Anthropic, or Google keys below. Ollama and AirLLM remain available but may be stopped on CPU-only VMs to save RAM.

Configure AI provider for automatic application analysis during registration. Supports local (Ollama) or cloud (OpenAI, Anthropic, Google, OpenAI-compatible) providers. API keys are stored server-side only — never sent to the browser.

Provider settings

Local SLMs on CPU may need 300–600 s for large codebases. Cloud APIs typically need 60–180 s.

Hybrid flow: Local SLM pre-summarizes your app's source code (fast, free, private), then Cloud LLM produces the structured analysis from that summary (accurate, fewer tokens = lower cost).

            CPU (%)
            
Cyan: Docker ≈% of host vCPUs. Magenta: CPU from /proc (host mount on Linux, or the container’s /proc on Docker Desktop = Linux VM). Orange + right axis: temperature from /sys.

            

          

          

            RAM (% · GB)
            
Y-axis is percent; hover tooltips and the summary line also show GB (Docker Σ vs MemTotal or Σ limits; magenta line ≈ avail or used vs MemTotal). Cyan: Docker Σ ÷ kernel MemTotal. Magenta: MemAvailable ÷ MemTotal (headroom) — not the same quantity as cyan. Without /proc, second line is Docker ÷ Σ cgroup limits.

            

          

          

            Network (Mb/s)
            
Cyan: Docker aggregate RX+TX Mb/s. Magenta: host/VM from /proc/net/dev when readable.

            

          

          

            IOPS & block I/O
            
Left: IOPS (often 0 on Docker Desktop). Right: Docker block throughput Mb/s

            

          

          

            Host CPU temperature
            

              Linux: max of /sys/class/thermal/*/temp (optional host mount DASHBOARD_HOST_SYS in dashboard.sh).
              macOS + Docker Desktop: optional brew install osx-cpu-temp. Starting LEco DevOps from ecosystem-stack/services/dashboard.sh installs a LaunchAgent that runs ecosystem-stack/scripts/macos-write-cpu-temp.sh every 30s; stopping/removing LEco DevOps removes it. On some Apple Silicon setups osx-cpu-temp reports 0°C; the writer then uses sudo powermetrics and maps thermal pressure to a proxy °C, plus load average so the chart moves under CPU load even when pressure stays Nominal. For unattended runs, passwordless sudo for /usr/bin/powermetrics may be needed. Deploy from the Mac so ~/.local-eco-host-metrics mounts into the container.
            

            

              No temperature samples yet — on Mac run the host temp script; on Linux ensure /sys thermal zones exist or mount host /sys.

              How dev stack builder works
              
    
                Pick a Stack preset (infrastructure, common bundles, or ready apps like WordPress / Magento) or choose components manually. Ready stacks can include sample demo content. LEco generates
                platform/dev-stacks/<id>/docker-compose.yml and runs it as project
                leco-devstack-<id> on its own Docker network. Databases stay internal;
                only routed HTTP services join lh-network. Bind a hosted app with
                platform.devStackId in leco.yaml (Hosted apps → Dev stack).
              
    
              
    
                Where configuration lives: each stack’s files are under
                platform/dev-stacks/<id>/ (compose, stack.yaml, optional
                varnish/ / nginx/). That is separate from
                hosting/app-available/ (registered hosted apps). HTTP routes are generated in
                hosting/traefik/20-dev-stacks.yml; the registry row is in
                config/leco-platform.yaml. Use Advanced — configuration & files on each stack card to view paths and edit files.

    Hosted apps
    
        
    
          LEco DevOps — per-app monitoring, compose logs, and lifecycle controls for entries in config/leco-registry.yaml (via leco-devops ecosystem-register and related commands). Control → Remove or Reset on a hosted stack runs leco-devops ecosystem-unregister, which runs docker compose down first (with -v on Reset), then Traefik strip, leco.local-cf.yaml cleanup, and registry row — same as Remove from ecosystem. If you stopped the stack only with docker compose down in a terminal, the app stays registered until you use Remove from ecosystem, the Routes tab, or leco-devops ecosystem-unregister. Set the Control token on the Control tab when DASHBOARD_CONTROL_TOKEN is enabled.
        
    
        
    
        
    
          Register application (generate YAML · save · register)
          
    
            
    
              Workflow: set path + app id, then Detect (scans
              docker-compose, wrangler.toml, and other config signals). Use Generate YAML
              to write leco.app.yaml + the localhost profile from that scan (or regenerate). Edit the text areas,
              then Save YAML to persist. Register stays disabled until both files exist on disk;
              it only runs leco-devops ecosystem-register (no silent overwrite). Generate/Save/Register require the
              control token when enabled. Read-only app roots materialize under hosting/app-available/<id>/
              (app id required). Path: repo-relative, wsp:…, or a host path when LEco DevOps was started with path hints.
              Label after Detect is guessed from package.json / Wrangler name / Compose top-level name when present (not from the folder name).
              URLs in Traefik (e.g. cv.lh) come from infrastructure.routing.entries[].hostname in your profile (leco.yaml), not from the registry slug; cloudflare.localCfPublicPrefix only shapes KV/R2/D1 hosts (cv-kv.lh, …). Use the Public URLs table below to edit every urls[] entry (dashboard, probes, docs); Write into profile YAML copies them into the textarea before Save, and Register applies the table to disk before ecosystem-register.
